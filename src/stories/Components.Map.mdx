# Components map

Płaska tabela wszystkich komponentów wystawionych z paczki `@juz/design-system`.
Każdy wiersz pokazuje: nazwę komponentu, warstwę architektury (Atom / Molecule /
Pattern / Layout / Screen), ścieżkę do pliku źródłowego w repo i link do strony
docs w tym Storybooku.

Tabela jest **generowana automatycznie** ze źródeł:

- lista komponentów: `src/index.ts` (publiczne re-exporty per warstwa),
- mapowanie komponent → story title: skan `src/stories/*.stories.tsx`.

Refresh po dodaniu nowego komponentu:

```bash
node scripts/regen-component-map.mjs
```

Skrypt nadpisuje tylko fragment między markerami `components-map:start` /
`components-map:end` poniżej — reszta strony jest pisana ręcznie i bezpieczna.

## Jak czytać warstwy

- **Atom** — pojedynczy primitive UI (`src/components/ui/`).
- **Molecule** — złożenie kilku atomów z domeną biznesową (`src/components/ds/`).
- **Pattern** — powtarzalny wzorzec pracy (DataList, Timeline, Charts). Pliki
  patternów również żyją w `src/components/ds/` razem z molecules.
- **Layout** — rama ekranu albo aplikacji (`src/components/layout/`).
- **Screen** — kompletny widok biznesowy w `src/stories/Screen.*.stories.tsx`.
  Screeny nie są re-exportowane jako komponenty (są przykładami do kopiowania),
  więc nie pojawiają się w tej tabeli — listę znajdziesz w sekcji **Screens** w
  navigation Storybooka.

## Czego brakuje w tej tabeli

- **Foundations** (kolory, typografia, tokeny, theme playground) — patrz sekcja
  **Foundations** w nawigacji.
- **Hooks** (`useSidebar`) — eksport publiczny, ale nie ma story.
- **Catalog overview** (`Atoms/Overview`, `Molecules/Overview`) — to przeglądowe
  story, a nie pojedyncze komponenty.
